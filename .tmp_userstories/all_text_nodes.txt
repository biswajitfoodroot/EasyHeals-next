User Stories — Phase 1
US-01
User Story
Symptom-to-Provider Search & Mapping
As a
patient or caregiver (any language, any city in India)
I want to
to describe my symptoms or health concern in natural language (English, Hindi, Tamil, Marathi, etc.) and immediately see matched doctors, hospitals, and clinics ranked by relevance and proximity
So that
I can find the right type of care without knowing the medical specialty name, saving time and reducing anxiety
Acceptance Criteria
Given I type 'seene mein dard' (Hindi for chest pain), the system extracts intent — specialty: Cardiology, language: Hindi — within 1.5 seconds.
Results show ONLY private hospitals/clinics; zero government hospitals ever appear.
Each result card shows: provider name, specialty tags, distance from detected/selected city, rating (if available), phone, and a 'Book / Call' CTA.
If confidence < 0.5, the system shows 'Did you mean: Cardiology?' with a one-tap confirm option before showing results.
Typing a direct specialty name (e.g. 'Orthopedic surgeon Mumbai') also returns correct results.
A language indicator badge (e.g. 🇮🇳 Hindi detected) is shown on the results header.
All 9 supported languages are handled: English, Hindi, Marathi, Tamil, Telugu, Kannada, Malayalam, Gujarati, Bengali.
Search latency p95 ≤ 1.5 seconds end-to-end.
UX Notes
Hero layout: AI chat input on the left, live results panel appearing on the right as user types (no page reload).
Skeleton loader cards appear instantly while API resolves — prevents perceived blank-screen wait.
Sticky intent badge at top of results ('Showing: Cardiologists · Pune · 42 results') collapses on scroll.
Mobile: full-screen search with bottom-sheet results sliding up after submit.
Voice input button (mic icon) for accessibility and low-literacy users.
Quick category pills below the search bar (Heart · Bones · Eyes · Child · Skin · Women) for browse-first users.
Entering only a city with no specialty shows 'Top hospitals near you' as a fallback, not an error.
API / Technical Notes
POST /api/search — Gemini 2.0 Flash extracts {specialty, city, symptoms, language, confidence}.
Dual query: Elasticsearch fuzzy match + Pinecone semantic vector search; results are rank-merged.
Search logs store SHA-256 hash of query only (HIPAA compliance — no raw PII stored).
Priority: P0 — Critical    Effort: High    Sprint: 1–2
US-02
User Story
Doctor–Hospital–Clinic Relationship Mapping & Navigation
As a
patient
I want to
to click any search result and see a full profile page that clearly shows which doctors practice at that hospital/clinic, and which hospital(s) a given doctor is affiliated with — with working map navigation
So that
I can make an informed choice and immediately get directions to reach the provider
Acceptance Criteria
Each Hospital Profile page shows: name, address, Google Maps embed + 'Get Directions' deep link, all affiliated doctors (name, specialty, schedule), departments, phone, photos, ratings.
Each Doctor Profile page shows: name, specialties, qualifications, affiliated hospital(s) with clickable links, consultation hours, and fee range.
Relationship links are bidirectional: clicking a doctor from a hospital page opens their profile; clicking a hospital from a doctor page opens the hospital profile.
'Get Directions' opens Google Maps / Apple Maps / OLA Maps with the exact coordinates pre-filled.
Nearby hospitals of the same specialty are shown in a 'Also nearby' section (≤ 5km radius).
All profile pages are server-side rendered (ISR, 1-hour revalidation) for SEO.
Each entity has a permanent URL: /hospital/[slug] and /doctor/[slug].
UX Notes
Profile pages use a tabbed layout: Overview | Doctors | Services | Reviews | Location.
Map is embedded inline (not a separate page); a 'Larger map' link opens full-screen.
Doctor cards on the hospital page show avatar, name, specialty chip, and a 'View Profile' button — not a wall of text.
Breadcrumb: Home > Cardiology > Pune > Apollo Hospital Pune — for orientation.
On mobile, the 'Call Now' and 'Get Directions' buttons are pinned to the bottom of the screen as a persistent action bar.
If a doctor has no photo, show a specialty-coloured monogram avatar (not a broken image).
API / Technical Notes
GET /api/hospitals/[slug] — returns hospital + affiliated doctors array.
GET /api/doctors/[slug] — returns doctor + affiliated hospitals array.
Mapping table: doctor_hospital_affiliations (doctorId, hospitalId, role, schedule).
Priority: P0 — Critical    Effort: Medium    Sprint: 2–3
US-03
User Story
Inline Editable Provider Information
As a
verified contributor or hospital admin
I want to
to click an 'Edit' or 'Suggest a correction' button next to any piece of information on a provider profile and submit an update directly from the same page
So that
the directory stays accurate without requiring me to fill out a separate form or email anyone
Acceptance Criteria
Every field on a Hospital/Doctor/Lab profile has a pencil (✏️) icon visible on hover (desktop) and always visible on mobile.
Clicking the icon opens an inline popover/drawer (not a new page) pre-filled with the current value.
Editable fields include: name, phone(s), address, specialties, working hours, website, photos, affiliated doctors/hospitals, consultation fee.
On submit, the system immediately shows 'Edit submitted — under review' (or 'Auto-approved' if AI confidence ≥ 80).
Edits from verified/trusted contributors (score history ≥ 85 avg) are auto-applied without manual review.
Anonymous users can submit edits but require CAPTCHA or phone OTP to prevent spam.
Edit history is accessible via a 'View edit history' link on every field — shows who changed what and when.
UX Notes
Edit popover opens at the field level, not page level — user sees their change in context.
For multi-value fields (phone list, specialties array), the popover shows an add/remove chip UI, not a raw text box.
After auto-approval, the UI updates optimistically in real time without a page refresh.
A 'Thank you for contributing' micro-toast (2 seconds) appears after every valid submission.
Edit history drawer uses a timeline layout: avatar + timestamp + old → new value for each change.
Rejected edits are shown as greyed-out with an AI reason (e.g. 'Phone number does not match city STD code').
API / Technical Notes
POST /api/contribute — accepts {hospitalId|doctorId, field, oldValue, newValue}.
Rate limit: 20 edits/hour per IP; batch edits (>20/hr) held for admin review.
Allowed fields whitelist enforced server-side to prevent injection.
Priority: P0 — Critical    Effort: Medium    Sprint: 2–3
US-04
User Story
AI-Powered Edit Validation with Continuous Learning
As a
platform operator
I want to
every submitted edit to be automatically scored by AI for validity, and for that scoring model to improve over time based on admin accept/reject decisions
So that
the moderation queue stays small, spam is blocked automatically, and the system gets smarter with every human decision
Acceptance Criteria
Every incoming edit receives an AI outlier score (0–100) within 2 seconds of submission.
Score ≥ 80 → auto-approved. Score 30–79 → queued for human review. Score < 30 → auto-rejected with reason shown to submitter.
Scoring considers: phone format matching city STD, address consistency, website domain plausibility, specialty name validity, duplicate detection.
Every human admin decision (approve / reject) is stored as a labelled training example with the original edit payload.
A nightly retraining job fine-tunes the scoring prompt or model using the accumulated labelled data.
Admin dashboard shows model accuracy metrics: precision, recall, false-positive rate over rolling 30-day window.
Threshold calibration (e.g. moving auto-approve from 80 to 85) is configurable from admin UI without code deploy.
Contributor reputation score is updated after each decision, influencing future threshold for that contributor.
UX Notes
Admin moderation card shows AI score as a colour-coded dial: green (≥70), amber (40–69), red (<40).
AI reason is shown in plain language: 'Mumbai hospital listed with Bengaluru STD code 080 — suspicious.'
Admin can override an auto-rejection with one click; override is logged as a training signal.
Contributor profile page shows their trust score, total edits, approval rate, and streak.
New contributors start in 'probation' mode — all edits go to human review until 10 consecutive approvals.
API / Technical Notes
scoreContribution() in lib/gemini.ts — returns {score, reason, flag}.
POST /api/admin/moderation PATCH — stores decision + triggers async training data append.
Nightly cron: scripts/retrain-validation.ts — aggregates labelled examples, updates prompt or fine-tuning dataset.
contributor_scores table tracks per-IP / per-account trust history.
Priority: P0 — Critical    Effort: High    Sprint: 3–4
US-05
User Story
Universal Search & Map for Doctor–Hospital–Lab Linking
As a
hospital admin or data moderator
I want to
a powerful search interface where I can find any doctor, hospital, or lab and create, edit, or remove the relationship mapping between them
So that
the directory accurately reflects real-world affiliations (e.g. Dr. Sharma consults at Apollo and Sahyadri on different days)
Acceptance Criteria
A global search bar (accessible from admin and public pages) returns mixed-type results: hospitals, doctors, labs — all in one dropdown, labelled by type.
Fuzzy matching handles typos: 'Appollo Hosptial' returns 'Apollo Hospital'.
From a Hospital page admin mode, an 'Add Doctor' flow searches the doctor index and creates an affiliation with schedule and role.
From a Doctor page admin mode, an 'Add Affiliation' flow searches hospitals and labs and creates the link.
Existing affiliations show 'Edit' (change schedule/role) and 'Remove' (soft-delete) buttons.
Lab ↔ Hospital mapping: a lab can be listed as associated with a hospital (e.g. 'In-house lab at XYZ Hospital').
All relationship changes are versioned and reversible from edit history.
The public map view (Google Maps embed) auto-pins all mapped providers in a given search area.
UX Notes
Global search uses a command-palette style (⌘K / Ctrl+K shortcut on desktop) with type icons: 🏥 Hospital, 👨‍⚕️ Doctor, 🔬 Lab.
Search results grouped by type in the dropdown: 'Hospitals (3)  ·  Doctors (7)  ·  Labs (2)'.
Affiliation creation is a drag-and-drop or chip-add UI — no raw ID entry.
A visual network graph (collapsible) on the Hospital/Doctor profile shows all mapped relationships at a glance.
Map view has layer toggles: show Hospitals | show Doctors | show Labs — independently togglable.
API / Technical Notes
GET /api/search/global?q=apollo&types=hospital,doctor,lab — returns mixed results.
POST /api/affiliations — {fromType, fromId, toType, toId, role, schedule}.
DELETE /api/affiliations/[id] — soft-delete (sets deletedAt, preserves history).
Priority: P1 — Important    Effort: Medium    Sprint: 4–5
Recommended Improvement Features
The following features are not in Phase 1 scope but are recommended for implementation in Phases 2–5 based on user research, competitive analysis, and the EasyHeals product vision.
IMP-01Personalisation
Personalised Health Journey
Remember past searches, preferred city, and health concerns (opt-in). Show 'Doctors you've viewed', 'Near your last search area', and proactive re-engagement: 'You searched for a Cardiologist 30 days ago — have you booked yet?'
Patient Impact: High     Build Effort: Medium
IMP-02Appointments
In-App Appointment Booking (Phase 3 Pull-Forward)
Add a 'Book Appointment' button to every doctor/hospital card. Even a simple 'Request callback' flow (name + phone + preferred time) dramatically reduces friction and generates leads for hospital partners before full booking goes live.
Patient Impact: Very High     Build Effort: Low
IMP-03Verified Reviews
Patient Review & Rating System with Verification
Allow patients to rate visits. Require OTP or appointment ID to submit a review (prevents fake reviews). AI flags reviews for abusive language, competitor spam, or suspiciously high volumes from one IP.
Patient Impact: High     Build Effort: Medium
IMP-04Real-time
Live Wait-Time & Availability Indicator
Hospitals that integrate via API can push real-time OPD wait time (e.g. '~25 min wait now'). For others, show 'Usually busy at this time' based on Google Popular Times data. Reduces wasted trips.
Patient Impact: High     Build Effort: High
IMP-05AI Triage
AI Pre-Triage Symptom Checker
Before showing results, offer a 3–5 question conversational triage: 'How long have you had this symptom? Is there fever? Any prior conditions?' Use answers to refine specialty match and show urgency level (Routine / Soon / Emergency). Reduces misdirected searches.
Patient Impact: Very High     Build Effort: High
IMP-06Accessibility
Voice Search & Regional Language TTS
Add voice input (Web Speech API / Whisper) for low-literacy users. Read out top 3 results in the user's detected language using Text-to-Speech. Critical for Tier 2/3 cities where typing in Hindi/Tamil is a barrier.
Patient Impact: High     Build Effort: Medium
IMP-07Trust
NABH / NABL Accreditation Badge
Scrape or source NABH (hospitals) and NABL (labs) accreditation status and show a verified badge. This is a high-trust signal for patients choosing between providers and strongly differentiates EasyHeals from generic directories.
Patient Impact: High     Build Effort: Low
IMP-08Data Quality
Automated Data Freshness Monitoring
Weekly automated checks: phone numbers dialled by a bot to verify they connect, websites pinged for 200 status. Stale/unreachable entries flagged with a 'Verify this listing' badge and queued for contributor update. Keeps directory self-maintaining.
Patient Impact: Medium     Build Effort: Medium
